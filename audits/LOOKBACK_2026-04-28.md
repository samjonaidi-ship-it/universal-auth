## Tier 2 — fix before Block 7 demo deploy

### L2: Circular test in security suite

**Where:** `test/security/02-timing-attack-resistance.test.ts:19`

```ts
function constantTimeEqual(a: string, b: string): boolean { ... }
```

**Problem:** the test defines `constantTimeEqual` locally and tests its own definition. The SDK doesn't export this helper. Test 1 ("matching vs first-char-mismatch indistinguishable") is testing the test, not the SDK. Test 2 (source-grep heuristic) IS real.

**Fix:** delete test 1, OR add `constantTimeEqual` as a real SDK export and test the export. Update THREAT_MODEL D7 row accordingly.

### L3: Soft assertion in IDB tamper test

**Where:** `test/security/04-idb-tamper.test.ts:87`

```ts
expect(corrupted).toBeGreaterThanOrEqual(0);
```

**Problem:** `>= 0` is always true, even when no byte arrays were found and no corruption happened. A green run with `corrupted === 0` provides zero defense.

**Fix:** change to `expect(corrupted).toBeGreaterThan(0)`. Now if the IDB schema changes and no `Uint8Array` exists, the test fails loudly.

---

### L4: Token replay test doesn't actually verify rotation

**Where:** `test/security/06-token-replay.test.ts` "rotation overwrites the old encrypted blob"

**Problem:** the test sets two sessions back-to-back and asserts `recordCount > 0`. This passes even if BOTH old + new sessions are stored side-by-side (which would be a real security bug — old refresh token still readable).

**Fix:** assert exactly 1 session record post-rotation, OR capture old blob bytes pre-rotation and verify they differ from post-rotation bytes.

---

### L5: AuthProvider-extras tests assert "fetch was called" only

**Where:** `test/unit/react/AuthProvider-extras.test.tsx`, two tests:
- "hydrates from /auth/v1/me when no initialSession given"
- "handles /auth/v1/me 401 → anonymous status (no crash)"

**Problem:** assert only `fetchSpy.toHaveBeenCalled()`. Don't verify the SDK's response handling (status state, identity context value, etc.). Tests would pass even if AuthProvider crashed silently after the fetch.

**Fix:** assert downstream state — e.g., for the 401 case, verify `screen.getByTestId('status').textContent === 'anonymous'`.

### L6: src/sw/index.ts excluded from coverage with NO test

**Where:** `vitest.config.ts` excludes `src/sw/**` with comment "covered by Playwright (Day 20-21)".

**Problem:** the Day 20-21 browser tests don't actually exercise the SW. They test sign-in, passkey, multi-tab — not SW registration, background-sync, or the cache-purge-on-logout logic. The 98 lines of real logic in `src/sw/index.ts` have ZERO test coverage today.

**Fix options:**
- (a) Write a Playwright test that registers the SW + exercises sync events + logout cache purge
- (b) Pull the testable logic into a separate module (e.g., `purgeCachesByPattern(patterns, caches)`) that can be unit-tested with a fake `caches` API
- (c) Accept the gap as a documented Block 7 must-do (demo deploy will exercise it manually)

**Recommended:** (b) is best because it validates the algorithm in CI. (a) is best if you also need the SW lifecycle integration test.

## Tier 3 — note, revisit at v1.1

### L10: `dist/meta.json` ships in the tarball

**Where:** `pnpm pack --dry-run` shows `dist/meta.json` (esbuild metafile) in the published artifact.

**Problem:** contains full filesystem paths from build (e.g., `node_modules/.pnpm/nanoid@5.1.9/...`) and internal `src/*.ts` filenames. Minor info disclosure — not catastrophic, but consumers don't need it.

**Fix:** add `dist/meta.json` to `.npmignore`, OR remove from build output, OR refine `files[]` to whitelist only `dist/esm` + `dist/types`.

---

### L12: Test setup swallows `InvalidStateError` but SDK never catches it in production

**Where:** `test/unit/setup.ts:39` adds `InvalidStateError` + `transaction is not active` to `SWALLOW_PATTERNS`. Justification: leaked `void emit()` after `__resetDbForTests`.

**Problem:** SDK source has zero `catch` handlers for `InvalidStateError`. If a real production bug closes the IDB mid-transaction, the SDK throws InvalidStateError → test setup silently swallows it → test passes. The filter is hiding a class of error the production code doesn't handle.

**Fix:** harden production path. Wrap `db.add()` / `db.put()` / `db.count()` in `event-reporter.ts` + `settings-sync.ts` with explicit `catch (e) { if (e.name === 'InvalidStateError') return; ... }` and emit a metrics counter. Then the production code is robust AND the test setup filter becomes legitimate (it's filtering noise from a path that's already handled gracefully).
